```

**Pipeline :**
```
Blueprint (YAML/dict) + Config (données)

    ↓ compile_model()

## Points d'attention

### 1. Gestion des dimensions

**Situation actuelle :**
- Ordre canonique fixe : `(E, T, F, C, Z, Y, X)`
- Les fonctions doivent connaître cet ordre et gérer le broadcasting manuellement
- `core_dims` utilisé pour la validation, pas pour transformer les données à l'exécution

**Risque :**
Une fonction avec `core_dims={"x": ["Z"]}` recevant `(C, Z, Y, X)` doit savoir que Z est à l'axe 1, pas 0.

**Recommandation :**
Documenter clairement la convention. Alternative future : mécanisme type `apply_ufunc` pour isoler automatiquement les `core_dims`.

### 2. Forçages volumineux

**Situation actuelle :**
- Métadonnées chargées en lazy (`chunks={}`)
- Données complètes en RAM pour l'exécution
- Chunking temporel (`batch_size`) mais pas spatial

**Impact :**
OK pour thèse (données < RAM). Limitation pour déploiement téraoctets.

**Recommandation :**
Reporter. Si nécessaire plus tard : streaming spatial ou intégration Dask.

### 3. Optimisation par gradient

**Situation actuelle :**
- `jax.lax.scan` différentiable ✓
- `trainable_params` dans `CompiledModel` ✓
- Pas de `GradientRunner` implémenté

**Prochaine étape :**
Implémenter l'optimisation. Schéma type :
```python
def loss(params, model, observations):
    model.parameters.update(params)
    _, outputs = runner.run()
    return jnp.mean((outputs["biomass"] - observations) ** 2)

grad_loss = jax.grad(loss)
```
