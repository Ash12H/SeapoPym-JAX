C-Coupler (pour Community Coupler) est un système de couplage de modèles climatiques et biogéochimiques conçu pour faciliter les échanges de données et la synchronisation entre différents sous-modèles (atmosphère, océan, glace de mer, biogéochimie, etc.). Il a été développé principalement par le Centre chinois de modélisation climatique (CMA / Tsinghua University) pour les besoins de la modélisation du système Terre.

1. Objectif général
   C-Coupler vise à standardiser et automatiser le couplage entre modèles hétérogènes (écrits en Fortran, C ou C++) sans qu’il soit nécessaire de modifier profondément leur code interne.
   Il agit comme une couche intermédiaire entre les sous-modèles, gérant :

Les échanges de champs (température, salinité, flux de CO₂, biomasse, etc.)

Les interpolations spatiales et temporelles nécessaires lorsque les grilles diffèrent

La synchronisation temporelle (chaque modèle peut avoir son propre pas de temps)

Le contrôle de la communication parallèle entre processus MPI

L’ordonnancement et la configuration du couplage via des fichiers XML ou scripts

2. Structure interne
   C-Coupler est organisé en plusieurs niveaux logiciels :

Niveau Rôle principal
Driver Initialise le système, charge les configurations et orchestre les échanges entre sous-modèles.
Decompositor Gère la décomposition parallèle (domaines locaux de calcul pour chaque modèle).
Remapper Effectue les interpolations entre grilles (sphère, curvilinéaire, etc.).
Timer & Synchronizer Coordonne les pas de temps et les moments d’échange.
Connector Gère les transferts de données entre deux sous-modèles donnés.
IO Server Permet la sauvegarde ou la lecture asynchrone des champs (I/O parallèle). 3. Fonctionnement
Le couplage repose sur une définition déclarative dans des fichiers de configuration :
l’utilisateur décrit quoi échanger, entre qui, quand et comment (interpolation, fréquence, unités).
Une fois le modèle lancé :

Chaque sous-modèle tourne sur son propre ensemble de processus MPI.

C-Coupler crée des canaux de communication entre ces ensembles.

À chaque pas de couplage, les champs définis sont exportés, interpolés et transférés vers les autres sous-modèles.

Un mécanisme de synchronisation garantit la cohérence temporelle.

4. Versions principales
   C-Coupler1 (2013) : première version, orientée modèles climatiques chinois (BCC-CSM).

C-Coupler2 (2016) : ajout du support multi-niveaux, I/O parallèle, regrillage flexible.

C-Coupler3 (2023) : refonte complète avec

meilleure extensibilité,

dynamique de couplage en ligne (modèles peuvent être ajoutés ou retirés pendant l’exécution),

gestion fine des dépendances temporelles,

compatibilité avec Earth System Modeling Framework (ESMF) et OASIS-MCT.

5. Positionnement par rapport à d’autres systèmes
   Système Langage / API Caractéristique principale
   OASIS-MCT Fortran / C Très utilisé en Europe (e.g. IPSL, EC-Earth) ; fiable mais configuration complexe.
   ESMF / NUOPC C / Fortran Cadre américain très complet mais lourd à intégrer.
   C-Coupler3 C++ / Fortran interface Couplage plus dynamique, architecture modulaire, meilleure gestion du parallélisme.
   C-Coupler se distingue par une forte automatisation du couplage et une flexibilité dans la configuration — sans nécessiter de recompiler tous les sous-modèles à chaque changement de scénario.

6. Usage typique
   Dans la pratique, il est employé pour assembler :

un modèle atmosphérique régional (e.g. CAM),

un modèle océanique (e.g. MOM ou NEMO),

un modèle biogéochimique ou d’écosystème (e.g. BFM),

un modèle de glace ou de surface continentale.

C-Coupler gère alors les échanges d’état (T, S, O₂, Corg, etc.) entre ces composantes.

Les workers communiquent via des messages Python (objets sérialisés) sans connaissance explicite de la structure spatiale des données.

Vous devrez définir un coordonnateur global (un acteur Ray maître) qui gère les événements d’échange, les décalages temporels et les latences réseau.

3. Performances et scalabilité
   Les coupleurs HPC sont conçus pour des machines massivement parallèles (jusqu’à plusieurs milliers de processus MPI), avec transferts mémoire-à-mémoire.

Ray est performant pour des tâches indépendantes ou faiblement couplées, mais le coût de sérialisation/desérialisation et les copies mémoire dans le Plasma Store deviennent un goulot d’étranglement si les messages portent sur des champs 3D volumineux (typiquement plusieurs Go).

Le couplage océan-biogéochimie demande souvent un taux d’échange élevé (chaque pas de temps ou sous-pas), donc les latences s’additionnent.

4. Description des échanges et traçabilité scientifique

Conclusion synthétique
Ton architecture Ray + messages peut devenir un coupleur robuste si tu ajoutes trois couches clés : (1) un registre de grilles + pré-calcul des poids (RegridManager), (2) un stockage partagé référencé (zarr/Plasma), (3) un Scheduler global assurant la cohérence temporelle. Avec ces briques, tu conserves la modularité, la résilience et l’élasticité de Ray tout en retrouvant les garanties qu’apportent les coupleurs classiques pour la gestion des grilles et la performance.
