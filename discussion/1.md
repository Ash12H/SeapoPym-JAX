1 — Principes & contraintes
État : stocké avec xarray (DataSet/DataArray) — variables indexées par dims spatiales (ex. y,x) et éventuellement depth/trait ; on garde hors-temps la dimension temporelle (on avance en place ou on écrit des snapshots).

Unités d’exécution (Units) : fonctions qui prennent l’état courant (ou des blocs d’état) + dt, t et qui produisent des modifications (nouvelle variable ou mise à jour d’une variable existante). Chaque unité déclare ses entrées/sorties.

Kernel : orchestrateur qui, pas de temps par pas de temps, exécute les unités dans un ordre (topologique si dépendances), gère les barrières (synchronisation) et déclenche le calcul en parallèle (via dask).

Parallélisme :

Si pas de transport / interactions spatiales, chaque cellule ou chunk peut être calculée indépendamment → dask map_blocks / apply_ufunc.

Si dépendances spatiales (transport), utiliser map_overlap (halo) ou une réduction globale; là il y a un point bloquant (synchronisation).

Numba : utile dans les kernels locaux / boucle cellulaire — on l’applique à la fonction qui opère sur numpy arrays. Avec xarray.apply_ufunc(..., dask='parallelized') on peut combiner dask + numba (en passant par numpy).

Extensibilité : décorateurs pour déclarer unités, plugin pour transport / couplage, hooks pour debug / checkpoint.

from typing import Callable, List, Dict, Any
import xarray as xr
import dask.array as da
import numpy as np
from dataclasses import dataclass, field
import inspect
try:
from numba import njit
except Exception:
njit = lambda f: f # fallback if numba not present

# -------------------------

# State wrapper

# -------------------------

class State:
def **init**(self, ds: xr.Dataset):
self.ds = ds

    def get(self, name: str):
        return self.ds[name]

    def set(self, name: str, array):
        self.ds[name] = (self.ds.dims.keys() and list(self.ds.dims.keys()) or ['y','x'], array) \
            if isinstance(array, np.ndarray) else array

    def snapshot(self):
        return self.ds.copy(deep=True)

# -------------------------

# Unit dataclass

# -------------------------

@dataclass
class Unit:
func: Callable
inputs: List[str]
outputs: List[str]
parallel: bool = True # embarrassingly parallel over spatial grid if True
needs_halo: int = 0 # number of halo cells required on each side
compiled: bool = False
metadata: Dict[str,Any] = field(default_factory=dict)

    def run(self, state: State, dt: float, t: float):
        # retrieve inputs as numpy arrays (or dask arrays)
        args = [state.ds[name].data for name in self.inputs]
        # call func(args..., dt, t)
        result = self.func(*args, dt=dt, t=t)
        if len(self.outputs) == 1:
            state.ds[self.outputs[0]] = (state.ds[self.inputs[0]].dims, result)
        else:
            for out, arr in zip(self.outputs, result):
                state.ds[out] = (state.ds[self.inputs[0]].dims, arr)

# -------------------------

# Kernel

# -------------------------

class Kernel:
def **init**(self, units: List[Unit], state: State, dt: float, total_steps: int, client=None):
self.units = units
self.state = state
self.dt = dt
self.total_steps = total_steps
self.client = client # optional dask client

    def step(self, step_index:int, t:float):
        # units executed in order; this is where you could do dependency-checking
        for unit in self.units:
            # if unit is dask-parallelizable, apply via xarray.apply_ufunc or compute and set
            if unit.parallel and isinstance(self.state.get(unit.inputs[0]).data, da.Array):
                # use apply_ufunc, support for multiple inputs/outputs is shown minimally
                xr_func = lambda *args, dt, t: unit.func(*args, dt=dt, t=t)
                xr.apply_ufunc(
                    xr_func,
                    *[self.state.get(name) for name in unit.inputs],
                    input_core_dims=[[]]*len(unit.inputs),
                    output_core_dims=[[]]*len(unit.outputs),
                    vectorize=True,
                    dask='parallelized',
                    kwargs={'dt': self.dt, 't': t},
                    output_dtypes=[float]*len(unit.outputs)
                ).compute()  # or return and assign — simplified
                # NOTE: above .compute() triggers; better is to set target variable with returned DataArray
            else:
                # fallback: run with concrete numpy arrays (synchronous)
                unit.run(self.state, self.dt, t)

    def run(self):
        t = 0.0
        for n in range(self.total_steps):
            self.step(n, t)
            t += self.dt

# Example units

# 1) mortality as function of temperature and biomass

@njit
def mortality*kernel(biomass_arr, temp_arr, dt, t): # shape: (y,x)
out = np.empty_like(biomass_arr)
for i in range(biomass_arr.shape[0]):
for j in range(biomass_arr.shape[1]):
b = biomass_arr[i,j]
temp = temp_arr[i,j]
rate = 0.01 + 0.001 * max(0, temp - 15.0) # toy relationship
out[i,j] = b \_ (1 - np.exp(-rate\*dt)) # biomass lost per dt
return out

# 2) update biomass after mortality and growth

@njit
def update_biomass_kernel(biomass_arr, growth_arr, mortality_arr, dt, t):
out = np.empty_like(biomass_arr)
for i in range(biomass_arr.shape[0]):
for j in range(biomass_arr.shape[1]):
b = biomass_arr[i,j]
g = growth_arr[i,j]
m = mortality_arr[i,j]
out[i,j] = b + g\*dt - m
return out

u_update = Unit(func=update_biomass_kernel, inputs=['biomass','growth','mortality'], outputs=['biomass'], parallel=True, compiled=True)

Remarques sur ce squelette :

J’ai simplifié xarray.apply_ufunc — en pratique on doit assigner le résultat de apply_ufunc à la variable de sortie et laisser dask gérer les tâches (pas .compute() à chaque unité sinon pas de parallélisme global).

Pour numba et dask : numba fonctionne sur numpy arrays. Avec dask, il faudra map_blocks sur les chunks et appeler la fonction numba à l’intérieur d’un worker (ou compiler pour chaque worker). Une option courante : vectorize avec xr.apply_ufunc(..., dask='parallelized') ou écrire wrappers qui reçoivent numpy.ndarray et sont njit-compilés.

4 — Gestion des dépendances temporelles et halo spatial
Temporal dependency (previous step) : soit tu lances les unités en lecture sur state qui contient la variable du pas précédent; si une unité a besoin du pas précédent de la même variable (ex. schéma multi-step), stocke la variable biomass_prev à chaque step (ou utilise un small ring buffer).

Dépendances spatiales :

Pas de transport : aucun halo, chunk indépendant.

Transport (ex. advection, diffusion) : utiliser dask.array.map_overlap pour appliquer un kernel local sur chaque chunk en lui fournissant le halo ; ou synchroniser globalement avant et après passage transport.

Exemple d’approche : découper en deux étapes par pas de temps

local updates (mortality, growth) — parfaitement parallèle.

transport step — applique map_overlap (nécessite halo N) ; après transport, sync/reassemble.

Si tu veux des performances maximales, éviter de transférer trop fréquemment entre workers ; regrouper unités qui n’ont pas de dépendances spatiales pour réduire communication.

5 — Stratégies pour combiner numba + xarray + dask correctement
Chunkning : chunk xarray/dask arrays selon y,x grid de taille raisonnable (ex. 64×64).

Use apply_ufunc : appeler fonctions numpy/numba avec xr.apply_ufunc(..., dask='parallelized', output_dtypes=...). Ta fonction doit accepter numpy arrays et renvoyer numpy arrays.

map_overlap pour transport : xr.apply_ufunc n’a pas directement map_overlap; pour cela travailler sur dask.array et map_overlap.

Avoid frequent .compute() : construire un graphe dask qui contient plusieurs unités puis compute() à intervalles plus larges (snapshot, outputs).

Compile numba on worker : si compilation lente, précompile avant dask submission (call once on small array) pour éviter overhead runtime.

6 — Orchestration plus avancée (topological ordering, dependencies explicites)
Chaque Unit peut déclarer inputs et outputs. Le Kernel peut construire un graphe de dépendances et détecter cycles, erreurs (ex : output non fourni).

Permettre unités conditionnelles (run_if) et événements (hook every N steps).

2. Architecture mono-processeur (simulation séquentielle d’agents)

a. Structure conceptuelle
class Cell:
def **init**(self, i, j, env, params, neighbors):
self.i = i
self.j = j
self.env = env # accède aux forçages, données externes

self.state = {'biomass': 1.0, 'temp': 15.0}

    def add_step(self, func):

class Grid:
def **init**(self, ny, nx, env, params):

3. Distribution parallèle : comment passer à l’échelle
   Quand ça marche en mono, tu as plusieurs chemins possibles pour la distribution.
   Tu peux la penser comme « chaque cellule = worker », mais il faut choisir le bon niveau de granularité pour ne pas exploser les coûts de communication.

Option A — Multi-threading / multiprocessing local
Utiliser concurrent.futures.ProcessPoolExecutor pour distribuer des blocs de cellules.

Chaque worker exécute son mini-grille (ex. 64x64), avance pas à pas, et échange les halos avec ses voisins via file de messages ou objets partagés.

Synchronisation après chaque pas de temps pour les flux entre blocs.

Option B — Dask
Chaque cellule (ou bloc) devient un Delayed Task dans le graphe Dask.

On peut exprimer les dépendances explicites : cell(t+1) dépend de neighbors(t).

Dask gère la planification, la persistance et le calcul distribué (local ou cluster).

Option C — Actor Model (Ray, multiprocessing.Pipe, asyncio)
Ray, ou même Python asyncio, permet une approche « agent » pure :

chaque Cell est un actor (process ou coroutine),

communication asynchrone : await neighbor.get_biomass(),

Ray gère la distribution sur cluster automatiquement.

Avantage : correspond exactement à ton schéma mental de « signal → réponse → reprise du calcul ».

Exemple minimal Ray :

import ray

@ray.remote
class CellActor:
def **init**(self, i, j, biomass):
self.i = i
self.j = j
self.biomass = biomass
self.neighbors = []

    def set_neighbors(self, neighbors):
        self.neighbors = neighbors

    def get_biomass(self):
        return self.biomass

    def step(self, dt):
        # lit les voisins
        neighbor_b = ray.get([n.get_biomass.remote() for n in self.neighbors])
        avg_b = sum(neighbor_b)/len(neighbor_b)
        self.biomass += 0.1*(avg_b - self.biomass)*dt

Chaque acteur Ray tourne indépendamment sur ton cluster, et le runtime gère les communications réseau.

5. Avantages de cette architecture
   ✅ Très intuitive : chaque cellule est une entité physique bien définie.

6. Points d’attention
   ⚠️ Overhead de communication si tu pousses le modèle « une cellule = un worker » littéralement. Il faut regrouper en blocs (patches).

   ⚠️ Performance Python pure : en mono-agent, tu risques d’être lent. D’où l’idée de numba + dask + ray pour accélérer.
