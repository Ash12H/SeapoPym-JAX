Ray est une bibliothèque Python open source conçue pour exécuter du code parallèle et distribué de manière simple, flexible et performante. Elle permet de passer d’un code local (monothread) à une exécution sur plusieurs cœurs, plusieurs machines, ou même un cluster complet — sans changer profondément la logique du programme.

🔍 Objectif et philosophie générale
L’objectif de Ray est de démocratiser le calcul distribué en offrant une abstraction simple :

« Le même code peut s’exécuter localement ou sur un cluster, sans changement majeur. »

Elle est souvent utilisée dans les domaines du machine learning, de la simulation, du traitement de données massives, ou de la modélisation scientifique, notamment lorsqu’il faut paralléliser un grand nombre de tâches indépendantes.

⚙️ Principes de base

1. Architecture et exécution
   Ray repose sur un modèle maître–travailleurs (head node / workers) :

Le Head node coordonne l’exécution, planifie les tâches et gère les objets.

Les Workers exécutent les tâches sur un ou plusieurs nœuds du cluster.

Un Object Store distribué (basé sur Plasma) conserve les objets partagés entre les tâches.

2. Abstraction principale : les tâches et les acteurs
   a) Les tâches Ray
   Ce sont des fonctions Python parallélisées :

import ray

ray.init()

@ray.remote
def f(x):
return x \* x

futures = [f.remote(i) for i in range(4)]
results = ray.get(futures)
print(results) # [0, 1, 4, 9]
Chaque appel à f.remote() crée une tâche indépendante pouvant être exécutée sur n’importe quel worker.

b) Les acteurs Ray
Un acteur est un objet Python vivant sur un worker, maintenant un état interne entre les appels :

@ray.remote
class Counter:
def **init**(self):
self.n = 0
def increment(self):
self.n += 1
return self.n

counter = Counter.remote()
print(ray.get(counter.increment.remote())) # 1
Les acteurs sont utiles pour modéliser des entités ayant une mémoire persistante, comme des agents ou des simulations locales.

3. Planification et tolérance aux pannes
   Ray gère automatiquement :

la planification dynamique des tâches selon les ressources disponibles ;

la reprise après panne (grâce au stockage distribué) ;

le passage automatique des données entre workers.

Plasma est le système de mémoire partagée développé à l’origine dans le projet Apache Arrow, et utilisé par Ray comme Object Store (magasin d’objets distribués).

Son rôle est central : il permet à plusieurs processus — potentiellement sur des nœuds différents — de partager efficacement des objets immuables (typiquement des tableaux NumPy, des DataFrames, ou des structures sérialisées) sans les copier en mémoire à chaque échange.

⚙️ Principe de fonctionnement

1. Mémoire partagée et immutabilité
   Les objets placés dans Plasma sont immuables : une fois stockés, ils ne peuvent plus être modifiés.

Cette contrainte permet d’éviter toute contention et de rendre le partage de mémoire thread-safe.

Chaque worker ou acteur accède à l’objet par un pointeur mémoire au lieu de le recopier.

2. Sérialisation via Apache Arrow
   Plasma repose sur le format Arrow, optimisé pour la sérialisation binaire de structures de données tabulaires ou vectorielles.

Cela permet d’échanger des objets entre langages (Python, Java, C++) sans coût de conversion.

3. Object Store distribué dans Ray
   Dans Ray, le Plasma Store est déployé :

Localement sur chaque nœud : chaque worker a accès au store de son nœud.

Avec synchronisation automatique : si une tâche a besoin d’un objet situé sur un autre nœud, Ray gère la récupération réseau et la mise en cache.

🧠 Illustration simplifiée
Une fonction distante produit un objet :

@ray.remote
def compute():
return np.arange(10)
obj_ref = compute.remote()
→ Le résultat est stocké dans le Plasma Object Store du nœud où la tâche s’est exécutée.

Une autre tâche consomme cet objet :

@ray.remote
def consume(x):
return x.sum()

total = ray.get(consume.remote(obj_ref))
→ Ray récupère l’objet depuis le store si besoin, sans duplication inutile.

⚙️ Avantages
Partage sans copie : réduction drastique du coût mémoire et CPU.

Accès rapide grâce à la mémoire partagée locale.

Interopérabilité avec de nombreux langages et frameworks via Arrow.

Sécurité et robustesse grâce à l’immuabilité.

⚠️ Limites
Objets non mutables : impossible de modifier un objet en place.

Sérialisation nécessaire : les objets Python complexes doivent être sérialisés (par pickle ou Arrow).

Gestion mémoire : le Plasma Store a une taille fixe, et un dépassement provoque des erreurs de type out of memory.

En résumé, Plasma est le cœur mémoire partagé de Ray, garantissant des échanges rapides, sûrs et efficaces entre processus distribués — un peu comme une “RAM commune” à tout le cluster, dédiée au calcul parallèle.

Je vais te décrire une approche raisonnée, adaptée à ton contexte de modélisation océanique, où les données de forçage (ex. température, courants, nutriments) sont souvent massives et multi-dimensionnelles.

⚙️ Objectif
L’idée est de rendre disponibles les champs de forçage à tous les workers :

sans dupliquer inutilement les données ;

en gardant une accès local rapide ;

tout en permettant à chaque worker de lire seulement ce dont il a besoin (spatialement et temporellement).

🧱 Étapes générales

1. Préparer les données dans un format optimisé pour l’accès distribué
   🔸 Format conseillé : Zarr
   Contrairement au NetCDF classique (monolithique et séquentiel), Zarr stocke les données :

en chunks indépendants (petits blocs) ;

accessibles en parallèle via un système de fichiers ou un stockage cloud (S3, GCS, etc.) ;

avec un accès direct par indexation (lazy loading possible).

💡 Tu peux convertir un NetCDF vers Zarr avec xarray :

import xarray as xr

ds = xr.open_dataset("forcing.nc")
ds.chunk({"time": 24, "lat": 100, "lon": 100}).to_zarr("forcing.zarr", mode="w") 2. Charger et distribuer via Ray + Xarray/Dask
Ray peut collaborer avec Dask et Xarray pour paralléliser le chargement ou la manipulation des données.

Exemple :
import ray
import xarray as xr

ray.init()

@ray.remote
def process_chunk(time_index):
ds = xr.open_zarr("forcing.zarr", chunks={})
subset = ds.isel(time=time_index) # traitement local (ex : interpolation ou calcul de gradient)
return subset.mean().compute()

results = ray.get([process_chunk.remote(t) for t in range(100)])
👉 Chaque worker ouvre le même store Zarr (lecture seule, lazy) et lit uniquement les blocs nécessaires.
Les données ne transitent pas inutilement par le réseau Ray : elles sont chargées localement à la demande.

3. Option : utilisation du Ray Object Store pour des sous-ensembles
   Si certains champs de forçage doivent être réutilisés plusieurs fois (ex. température moyenne, masque bathymétrique), il peut être intéressant de les placer explicitement dans le Plasma Object Store pour éviter de les relire depuis le disque.

import ray

@ray.remote
def load_static_field():
ds = xr.open_zarr("forcing.zarr", chunks={})
return ds["bathymetry"].load() # Chargé en mémoire

static_field_ref = ray.put(ray.get(load_static_field.remote()))
Chaque worker peut ensuite récupérer ce champ directement en mémoire partagée :

@ray.remote
def run_model(cell_id, bathy_ref):
bathy = ray.get(bathy_ref) # calcul utilisant bathy
⚠️ Il faut cependant limiter la taille de ce qui est stocké dans le Plasma Store (quelques Go maximum par nœud), sous peine de saturation.

4. Optimisation des accès
   Chunking adapté : les chunks Zarr doivent correspondre à ton schéma d’accès (par ex. blocs temporels ou spatiaux cohérents avec les tâches Ray).

Locality awareness : si possible, placer les stores Zarr sur le même nœud que les workers traitant les mêmes zones.

Cache disque ou mémoire : Xarray/Dask peuvent garder temporairement les chunks les plus utilisés.
